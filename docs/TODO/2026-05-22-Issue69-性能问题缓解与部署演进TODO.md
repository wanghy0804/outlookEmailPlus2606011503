## Phase 3：gevent 单 worker 评估

**目标**：在不立刻拆 scheduler 的前提下，评估部署层最小缓解方案。

### Task 3.2：兼容性风险清单
- [x] APScheduler BackgroundScheduler：🔴 高风险（线程 → greenlet 替换）
- [x] ThreadPoolExecutor：🟡 中风险（`as_completed` 可能失序）
- [x] threading.Lock × 3：🟡 中风险（替换为 BoundedSemaphore）
- [x] SQLite WAL 共享：🟡 中风险
- [x] SSE/yield 流式：🟢 低风险
- [x] urllib.request：🟢 低风险

## Phase 4：scheduler 拆分设计

**目标**：从根上去掉"Web 进程 = scheduler 宿主"的耦合。

### Task 4.1：列出当前 scheduler job 清单
- [x] `scheduler_heartbeat` — 60s 写入心跳
- [x] `email_notification_job` — 动态间隔，统一通知分发 (Email/Webhook)
- [x] `external_probe_poll` — 5s，异步探测 pending + 清理过期
- [x] `pool_expire_stale_claims` — 60s，回收过期 claimed
- [x] `pool_recover_cooldown` — 300s，恢复 cooldown 账号
- [x] `token_refresh` — Cron 定时刷新 OAuth token

### Task 4.2：决定拆分形态
- [x] 方案 A：独立 Python 进程 + 进程管理器（推荐）
- [x] 方案 B：Docker sidecar 容器（备选）
- [x] 方案 C：轻量级双进程 shell 脚本（快速验证）
- [x] 推荐顺序：先 C 验证 → 稳定后用 A

### Task 4.3：定义边界与依赖
- [x] Web 进程：`create_app(autostart_scheduler=False)`
- [x] Scheduler 进程：持有 Flask app context，不绑定 WSGI
- [x] 共享资源：SQLite WAL + distributed_locks 保护
- [x] Flask app context 依赖分析：heartbeat/pool 无需；notification/probe/token_refresh 需保留

### Task 4.4：多 worker 前置清单
